# WebSocket项目实现梳理

你上传的这 4 个文件，完整实现了一套基于 FastAPI 的实时长连接消息推送系统，用来解决传统 HTTP 接口无法实现「服务器主动向客户端推送实时消息」的问题。

下面我帮你拆解整个实现的逻辑、每个模块的作用，以及 WebSocket 在这里具体发挥的价值。


---
## 一、整体架构概览

整个 WebSocket 系统分为 4 个核心模块，各司其职：

|文件|核心作用
|---|---|
|ws.py|对外暴露 WebSocket 接入接口，处理连接生命周期（建立 / 心跳 / 断开）
|ws_service.py|连接认证工具，验证客户端身份，确保只有合法用户能建立连接
|events.py|定义所有支持的实时事件类型，统一消息格式规范
|manager.py|全局连接管理器，维护所有在线用户的长连接，提供消息推送能力  

---
## 二、核心模块详细说明

### 1. 连接接入与生命周期管理 (ws.py)

这是整个系统的入口，对外提供了 /ws 这个 WebSocket 接口，处理客户端的连接全流程：

#### 连接建立流程

1. 客户端发起 WebSocket 连接请求，在请求参数中携带身份 token

2. 服务端调用 parse_user_id_from_token 解析 token，验证用户身份

  > 如果 token 无效 / 缺失，直接拒绝连接（关闭码 4401，代表未授权）

3. 认证通过后，调用连接管理器的 connect 方法，正式接受连接，并把这个长连接注册到管理器中

4. 连接建立完成后，全局广播「用户上线」事件，通知所有在线用户该用户已上线

#### 连接保活与消息处理

连接建立后，服务端进入循环，持续监听客户端发来的消息：

- 目前实现了心跳保活机制：如果客户端发来 ping 消息，服务端立刻返回 pong 心跳响应

  - 这个机制用来维持长连接不被中间的网络设备（比如防火墙）自动断开，同时检测连接是否存活

- 后续可以扩展支持更多客户端到服务端的消息类型（比如客户端主动发送指令）

#### 连接断开处理

当客户端主动断开连接，或者网络异常导致连接中断时：

1. 调用管理器的 disconnect 方法，从全局连接列表中移除这个失效的连接

2. 全局广播「用户下线」事件，通知所有在线用户该用户已离线


---
### 2. 身份认证 (ws_service.py)

这个模块非常简单，是连接的「安全网关」：

- 调用项目已有的 decode_access_token 工具，解析客户端传来的 JWT token

- 从 token 中提取用户 ID（sub 字段），验证 token 的合法性

- 只有合法的登录用户，才能成功建立 WebSocket 长连接，防止匿名用户占用连接资源


---
### 3. 事件类型定义 (events.py)

这个文件定义了所有支持的实时事件类型，用来统一前后端的消息格式，让客户端可以根据事件类型来处理不同的实时通知：
```python
# 基础连接事件
USER_ONLINE = "user_online"       # 用户上线
USER_OFFLINE = "user_offline"     # 用户下线
HEARTBEAT = "heartbeat"           # 心跳响应

# 房间相关事件
ROOM_UPDATED = "room_updated"     # 房间信息更新
ROOM_STARTED = "room_started"     # 房间游戏开始

# 游戏相关事件
GAME_WIN = "game_win"             # 游戏胜利通知
GAME_DRAWN = "game_drawn"         # 游戏平局
GAME_ROUND_END = "game_round_end" # 单轮游戏结束
# ... 其他游戏状态事件

# 任务相关事件
TASK_CREATED = "task_created"     # 新任务创建
TASK_UPDATED = "task_updated"     # 任务信息更新
TASK_STATUS_CHANGED = "task_status_changed" # 任务状态变更
```
这些事件就是 WebSocket 要推送的核心业务消息，比如：

- 当你完成了一个任务，服务器不需要等你轮询，直接通过 WebSocket 把 TASK_STATUS_CHANGED 事件推给你

- 当游戏对手出完牌，服务器立刻把 GAME_ROUND_END 推给你，你就能实时看到结果


---
### 4. 全局连接管理器 (manager.py)

这是整个系统的核心大脑，负责维护所有在线用户的长连接，提供消息推送的能力：

#### 连接存储结构

它用一个字典 user_sockets 来维护用户和连接的映射：
```Python
# key: 用户ID
# value: 该用户的所有 WebSocket 连接集合（支持一个用户多端同时在线，比如网页+APP）
self.user_sockets: dict[int, set[WebSocket]] = defaultdict(set)
```
核心能力

1. 连接管理：

  - connect：接受连接，把新连接添加到对应用户的连接集合中

  - disconnect：移除失效的连接，如果用户所有连接都断开了，就清理掉这个用户的记录

2. 定向消息推送：send_to_user

  - 给指定用户推送消息，比如：

    - 你的任务更新了，只需要推送给你自己

    - 游戏对手的操作，只需要推给参与游戏的两个玩家

  - 推送时会自动检测连接是否存活，如果发现某个连接已经失效，会自动清理掉

3. 全局广播：broadcast

  - 给所有在线用户推送消息，比如：

    - 用户上线 / 下线的全局通知

    - 系统的全局公告

## 三、WebSocket 在你的项目中发挥的作用

传统的 HTTP 接口是「请求 - 响应」模式，只能客户端主动问服务器，服务器才能回答，无法主动给客户端发消息。

而你的项目里，有很多需要实时通知的场景：

- 游戏对战：对手的操作、游戏结果需要立刻让你看到

- 任务通知：新任务、任务完成需要立刻提醒你

- 在线状态：好友的上下线状态需要实时同步

如果不用 WebSocket，你只能用「轮询」的方式：客户端每隔几秒就调用一次接口，问服务器「有没有新消息？」，这会：

1. 浪费大量的网络请求和服务器资源

2. 消息有延迟，无法做到实时

3. 体验很差

而用了 WebSocket 之后：

1. 只需要建立一次长连接，之后就可以持续通信

2. 服务器可以主动把消息推给客户端，消息几乎没有延迟

3. 节省了大量的轮询请求，降低了服务器压力

4. 实现了真正的实时交互体验

## 四、典型的业务使用流程

举个例子，当一场游戏结束时，整个流程是这样的：

1. 游戏服务检测到游戏结束，生成了胜利结果

2. 调用 WSManager.send_to_user(winner_id, {"type": GAME_WIN, "data": 胜利信息})

3. 管理器找到 winner_id 对应的所有 WebSocket 连接，把这个消息推过去

4. 客户端收到这个事件，立刻弹出胜利提示，更新界面，整个过程不需要客户端做任何轮询，完全实时！
